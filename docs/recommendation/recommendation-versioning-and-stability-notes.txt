## 1. `2.9.1 → 2.9.2` 방식은 적절합니다

Phase 3가 단순 후속 기능이 아니라, 별도의 데이터 검증과 플랫폼화 자료를 전제로 한다면 지금 당장 `3.0`으로 넘어가지 않는 것이 맞습니다.

추천 버전 의미는 다음과 같습니다.

| 버전      | 의미                           |
| ------- | ---------------------------- |
| `2.9.0` | 최초 완성된 랭킹·재생·평가 구조           |
| `2.9.1` | 기존 계약을 유지하는 계산 검증 보완         |
| `2.9.2` | 데이터 품질·무결성 검증 추가             |
| `2.9.3` | 실제 저장 계층 연결 준비               |
| `2.9.4` | API·플랫폼 통합 계약                |
| `2.9.x` | Phase 3 진입 전 안정화             |
| `3.0.0` | 신뢰 저장소와 실제 플랫폼 실행 경로가 연결된 시점 |

중요한 기준은 숫자가 아니라 **호환성**입니다.

* 기존 입력과 출력 계약이 유지됨 → patch 버전
* 선택적 필드나 독립 모듈이 추가됨 → minor 성격
* 의미나 데이터 모델이 깨짐 → major 전환

프로젝트 내부에서는 `2.9.1`, `2.9.2` 식으로 운영해도 문제없습니다.

# 2. 현재 구조가 쉽게 무너지지 않는 이유

지금 패키지에는 이미 몇 가지 강력한 방어선이 있습니다.

### 순수 함수 중심

DB, 네트워크, 환경변수, 현재 시각에 직접 의존하지 않습니다.

따라서 저장소나 플랫폼 계층이 바뀌더라도 알고리즘 계산 자체는 영향을 덜 받습니다.

### 정책 버전 분리

가중치와 알고리즘 행동이 정책 객체로 분리돼 있습니다.

기존 정책을 수정하지 않고 다음처럼 새 정책을 추가할 수 있습니다.

```text
ranking-policy-v1
ranking-policy-v2
diversity-policy-v1
diversity-policy-v2
```

### 재생 구조

과거 입력과 정책을 보존하면 새 코드를 넣은 후에도 기존 결과가 달라졌는지 즉시 검출할 수 있습니다.

### 강한 테스트

963개 테스트는 기능 확인뿐만 아니라 **현재 행동을 고정하는 안전망** 역할을 합니다.

### provenance 분리

개인화 결과와 탐색 결과의 발생 원인을 분리해 두었기 때문에, 향후 계산 방식이 늘어나도 결과의 출처를 추적할 수 있습니다.

즉 현재 시스템은 “변경에 강한 구조”가 맞습니다.

# 3. 앞으로 위험한 것은 난도가 아니라 변경 방식입니다

P0, P1, P2 항목이 어려워 보이지만, 모두 한꺼번에 기존 알고리즘 내부에 집어넣을 필요는 없습니다.

예를 들어 스냅샷 해시를 추가한다고 해서 랭킹 함수에 SHA-256 코드를 집어넣으면 안 됩니다.

```text
잘못된 방식

rankCandidates()
 ├─ DB 조회
 ├─ JSON 정규화
 ├─ SHA-256 생성
 ├─ 점수 계산
 ├─ 로그 저장
 └─ 평가 실행
```

이렇게 되면 결합도가 폭발합니다.

올바른 방식은 다음입니다.

```text
플랫폼 계층
 ├─ 데이터 조회
 ├─ canonical snapshot 생성
 ├─ hash 검증
 └─ 검증된 입력 전달
          ↓
알고리즘 코어
 ├─ score
 ├─ rank
 ├─ diversity
 └─ exploration
          ↓
감사·평가 계층
 ├─ exposure 저장
 ├─ replay
 ├─ outcome attribution
 └─ policy evaluation
```

각 계층의 책임을 유지하면 됩니다.

# 4. 기존 구조를 보존하는 절대 규칙

앞으로는 아래 다섯 가지를 사실상 헌법으로 두는 것이 좋습니다.

## 규칙 1. 기존 정책은 수정하지 않고 새 버전을 추가

나쁜 예:

```ts
export const rankingPolicyV1 = {
  interestWeight: 0.5, // 기존 0.4에서 수정
};
```

좋은 예:

```ts
export const rankingPolicyV1 = {
  interestWeight: 0.4,
};

export const rankingPolicyV2 = {
  interestWeight: 0.5,
};
```

과거 replay는 반드시 V1을 계속 사용할 수 있어야 합니다.

---

## 규칙 2. 새 필드는 기본적으로 optional 또는 새 타입으로 추가

기존 타입에 필수 필드를 갑자기 넣으면 모든 과거 스냅샷이 깨집니다.

```ts
type CandidateV1 = {
  entityId: string;
};
```

다음 버전에서는 가능한 한:

```ts
type CandidateV2 = CandidateV1 & {
  snapshotHash?: string;
};
```

또는 명확하게 버전을 분리합니다.

```ts
type CandidateSnapshotV1 = ...
type CandidateSnapshotV2 = ...
```

---

## 규칙 3. 계산 코어에는 저장소 코드를 넣지 않기

코어 함수는 계속 다음 형태를 유지해야 합니다.

```ts
input -> deterministic calculation -> output
```

DB repository, ORM entity, HTTP request, Redis, Supabase 객체를 코어 타입으로 직접 전달하면 안 됩니다.

플랫폼 데이터를 먼저 도메인 입력으로 변환해야 합니다.

```ts
const algorithmInput = mapDatabaseRowToCandidateSnapshot(row);
```

---

## 규칙 4. 과거 결과는 수정하지 않고 새 평가 결과를 추가

과거 evaluation row를 업데이트하는 것이 아니라 새 실행 결과를 append해야 합니다.

```text
evaluation_run_001
evaluation_run_002
evaluation_run_003
```

과거 실행 결과가 바뀌면 감사와 비교의 의미가 사라집니다.

---

## 규칙 5. 새 기능에는 반드시 호환성 게이트를 둠

새 기능을 병합하기 전에 최소한 다음을 확인해야 합니다.

```text
기존 전체 테스트 PASS
기존 golden replay 결과 불변
새 정책만 의도한 차이 발생
입력 객체 불변
동일 seed 결과 동일
과거 policy version 실행 가능
```

---

# 5. P0부터 P2까지 구조를 안 깨고 넣는 방법

## P0-1. 신뢰 가능한 스냅샷

랭커를 수정할 필요가 거의 없습니다.

새 모듈을 별도로 추가하면 됩니다.

```text
snapshot/
 ├─ canonicalize-snapshot.ts
 ├─ hash-snapshot.ts
 ├─ verify-snapshot-hash.ts
 └─ snapshot-contract.ts
```

랭커는 검증된 스냅샷만 받습니다.

---

## P0-2. DB와 이벤트 저장

도메인 패키지 외부에 adapter를 만듭니다.

```text
application/
 ├─ recommendation-service
 ├─ evaluation-service
 └─ exposure-service

infrastructure/
 ├─ recommendation-repository
 ├─ exposure-repository
 ├─ outcome-repository
 └─ snapshot-repository
```

현재 알고리즘 코드를 DB 엔티티에 맞춰 수정하는 것이 아니라, repository에서 도메인 입력으로 변환합니다.

---

## P0-3. 100개 후보 제한

현재 랭커 앞에 retrieval 계층을 추가하면 됩니다.

```text
전체 콘텐츠
  ↓
eligibility query
  ↓
candidate retrieval 500개
  ↓
pre-ranking 100개
  ↓
현재 알고리즘
  ↓
최종 20개
```

현재 랭커는 그대로 유지할 수 있습니다.

---

## P1. 학습형 또는 세그먼트형 가중치

기존 scoring 함수를 폐기할 필요가 없습니다.

새로운 정책 선택기를 앞에 둡니다.

```ts
const policy = selectRankingPolicy({
  userSegment,
  surface,
  sessionContext,
});

return rankCandidates({
  ...input,
  policy,
});
```

가중치가 ML 모델에서 나오더라도 최종 출력이 기존 정책 계약과 같으면 코어는 보존됩니다.

---

## P1. 다양성 동적 조정

다양성 알고리즘을 다시 쓰는 대신 정책만 선택합니다.

```text
home-feed-diversity-v1
search-results-diversity-v1
place-detail-diversity-v1
trip-planner-diversity-v1
```

같은 엔진에 다른 설정을 공급하면 됩니다.

---

## P2. 통계 평가

현재 deterministic evaluator 뒤에 통계 계층을 추가합니다.

```text
현재 evaluation result
  ↓
statistical analysis
 ├─ confidence interval
 ├─ bootstrap
 ├─ effect size
 └─ segment comparison
  ↓
release evidence report
```

기존 평가 판정은 구조적 무결성 게이트로 유지하고, 통계 판정은 별도 게이트로 둡니다.

```text
Gate A: 계산·데이터 구조 정상 여부
Gate B: 증거량 충분 여부
Gate C: 성과 개선 여부
Gate D: 운영 배포 승인
```

이렇게 해야 구조 검증과 성과 검증이 섞이지 않습니다.

# 6. 실제로 구조가 무너지는 시나리오

다음 행동만 피하면 됩니다.

* 기존 정책 상수를 덮어쓰기
* 과거 스냅샷 타입을 강제 마이그레이션
* 랭킹 함수에서 DB를 직접 호출
* 랭킹 결과를 저장하면서 객체를 변형
* 평가 함수 안에서 현재 시간을 읽기
* seed가 있는데도 내부에서 난수를 생성
* 기존 결과가 달라졌는데 테스트를 새 결과에 맞춰 수정
* 정책 버전을 `latest` 하나로만 관리
* DB 행을 그대로 알고리즘 타입으로 사용
* 해시, 저장, 계산, 평가를 하나의 서비스에 몰아넣기

특히 가장 위험한 것은 이것입니다.

> **테스트 실패를 “기능이 발전했으니 당연하다”면서 기존 기대값을 일괄 수정하는 것**

기대값이 바뀌었다면 반드시 다음 중 하나여야 합니다.

1. 새 정책 버전에서만 변경되었다.
2. 기존 구현에 실제 버그가 있었다.
3. 명시적인 breaking change로 승인되었다.

그 외에는 회귀입니다.

# 7. 지환님이 모든 수학과 구현을 직접 이해해야 하는가

그럴 필요는 없습니다.

다만 프로젝트의 통제권을 유지하려면 **수식 자체보다 계약과 판정 기준**을 이해해야 합니다.

지환님이 직접 판단해야 할 것은 다음 정도입니다.

| 영역    | 직접 통제할 내용                  |
| ----- | -------------------------- |
| 추천 목적 | 무엇을 좋은 추천으로 볼 것인가          |
| 정책    | 관심·상황·다양성·탐색 중 무엇을 우선할 것인가 |
| 금지 조건 | 어떤 후보는 절대 추천하면 안 되는가       |
| 검증    | 어떤 변화까지 허용할 것인가            |
| 출시    | 어떤 근거가 있어야 새 정책을 적용할 것인가   |
| 감사    | 과거 결과를 재현할 수 있어야 하는가       |

반면 아래는 AI에게 충분히 위임할 수 있습니다.

* 타입 설계
* 검증 함수 구현
* canonical serialization
* 해시 처리
* 통계 계산
* property-based test
* mutation test
* fixture 생성
* DB migration
* API adapter
* 성능 벤치마크

즉 지식 임계점을 넘었다는 것은 프로젝트를 중단해야 한다는 뜻이 아니라, **직접 코딩하는 역할에서 시스템 계약을 통제하는 역할로 올라가야 한다는 뜻**입니다.

# 8. 기술적 한도를 두지 않을 때 필요한 운영 방식

한도를 없애는 것은 가능합니다. 대신 변경 절차는 더 엄격해야 합니다.

추천 흐름은 다음입니다.

```text
1. 요구사항 정의
2. 현재 계약 영향 분석
3. 새 모듈 또는 새 정책 버전 설계
4. 구현
5. 신규 테스트 작성
6. 기존 963개 회귀 테스트
7. golden replay
8. mutation/property test
9. 성능 비교
10. 독립 리뷰
11. 버전 고정
12. 변경 기록
```

모든 기능에서 이 전체 절차를 적용하면 속도가 너무 느려질 수 있으므로 위험도에 따라 나누면 됩니다.

### 낮은 위험

* 문서
* 설명 메시지
* 내부 이름 개선
* 새 fixture

일반 테스트만 수행.

### 중간 위험

* 새 계산 컴포넌트
* 새 정책
* 새 평가 지표

전체 회귀 + golden replay.

### 높은 위험

* 귀속 방식 변경
* ID 규칙 변경
* snapshot canonicalization
* 정책 선택 로직
* 저장 무결성
* 배포 판정

전체 회귀 + 독립 검증 + 호환성 보고서.

# 최종 판단

**현재 구조를 망가뜨리지 않고 충분히 진행할 수 있습니다.**

오히려 현재 패키지가 순수 계산, 명시적 정책, 결정론, 버전 바인딩, replay를 이미 갖추고 있기 때문에 이후 복잡한 기능을 붙일 기반은 상당히 좋습니다.

다만 앞으로의 원칙은 명확해야 합니다.

> **수정이 아니라 추가, 교체가 아니라 버전 병렬화, 통합이 아니라 계층 분리, 추측이 아니라 회귀 증명.**

이 원칙만 유지하면 기술적 한도를 두지 않아도 됩니다. 문제는 알고리즘 난도가 아니라, AI가 생성한 대규모 변경을 아무 검증 없이 기존 코어에 섞는 경우입니다. 그 부분만 차단하면 이 프로젝트는 현재 수준보다 훨씬 더 멀리 확장할 수 있습니다.
